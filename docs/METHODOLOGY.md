# Methodology — Marchantia HH-suite v3 profile database (v7.1)

Citation-ready methods description for the *Marchantia polymorpha* HH-suite
profile database. Drop into a paper, grant, or report; the wording is intended
to be readable as a single Methods section while remaining a complete spec.

---

## Brief (1-paragraph version)

A homology-search database for *Marchantia polymorpha* (Tak-1, genome release
v7.1, n = 18,007 primary-isoform protein sequences) was built using HH-suite3
(v3.3.0, Steinegger et al. 2019). For each protein, a multiple-sequence
alignment (MSA) was generated by two iterations of `hhblits` against the
UniRef30 sequence/profile database (release 2023_02; Mirdita et al. 2017)
with `-cov 0 -qid 0 -maxfilt 100000 -diff inf -id 100 -e 1e-3 -n 2`. The
resulting A3M MSAs were converted to hidden-Markov-model (HMM) profiles with
`hhmake -M a3m`. Per-protein A3M and HMM files were packed into HH-suite's
`ffindex`/`ffdata` format with `ffindex_build`, and prefilter column-state
sequences (cs219) were derived from the a3m ffindex via `cstranslate -f
-x 0.3 -c 4 -I a3m`. The cs219 index was size-sorted descending and all three
indices (a3m, hhm, cs219) were reordered to match (load-balancer requirement).
The full database (a3m: 213 GB, hhm: 1.1 GB, cs219: 7.4 MB; 18,007 entries
per index) is searchable with `hhsearch`/`hhblits`. The pipeline is
orchestrated by Snakemake (v9.20.0; Mölder et al. 2021) on a SLURM cluster
(bwHPC Helix) with length-tiered job batching, atomic per-protein writes, and
idempotent restart. Source code: https://github.com/Papareddy/hhdb_marchantia.

## Full methods (paper-length)

### Input proteome
The primary-isoform protein set from the *Marchantia polymorpha* Tak-1 v7.1
genome assembly (`MpTak_v7.1.protein.primary.fa`, n = 18,007 sequences;
marchantia.info) was used as input. Length distribution: median 351 AA,
range 22–9,376 AA. No sequence filtering, masking, or pre-clustering was
applied — every primary-isoform protein has a corresponding profile in the
final DB (100 % proteome coverage).

### MSA construction (hhblits)
For each query sequence, two iterations of `hhblits` (HH-suite v3.3.0,
Steinegger et al. 2019) were run against the UniRef30 sequence/profile
database (release 2023_02, ~67 M clustered UniRef30 sequences; Mirdita et
al. 2017) with the following flags:

```
hhblits -i {query}.fa -d UniRef30_2023_02 -oa3m {id}.a3m \
  -n 2 -e 1e-3 -cov 0 -qid 0 -maxfilt 100000 -diff inf -id 100 -cpu 4
```

The non-default flags (`-cov 0 -qid 0 -maxfilt 100000 -diff inf -id 100`)
follow the HH-suite wiki's recommendation for **build-time** profiles: keep
MSAs **broad and unfiltered** so that filtering decisions can be made at
search time rather than baked into the DB. This trades disk space (213 GB
a3m total) for sensitivity.

### Profile construction (hhmake)
A3M MSAs were converted to HHM profiles via:

```
hhmake -i {id}.a3m -o {id}.hhm -M a3m
```

The `-M a3m` match-state assignment was used (rather than `-M first` or
`-M 50`) because `hhblits` v3.3.0 emits A3M files whose match-state
encoding occasionally triggers a column-count error (`"Compress: sequences
do not all have the same number of columns"`) when recomputed downstream;
`-M a3m` trusts the A3M's existing match-state encoding (uppercase = match,
lowercase = insert) and bypasses this recomputation.

### ffindex packing
Per-protein `.a3m` and `.hhm` files were assembled into HH-suite's
`ffindex`/`ffdata` format using `ffindex_build` (Hauser et al. 2016):

```
ffindex_build -s db_a3m.ffdata db_a3m.ffindex stage/a3m/
ffindex_build -s db_hhm.ffdata db_hhm.ffindex stage/hhm/
```

Symlinks in `stage/a3m/` and `stage/hhm/` were named bare protein IDs (no
file extension), so all three eventual indices (a3m, hhm, cs219) share the
same key set — this is a requirement of `hhsearch`'s key-lookup logic and a
non-obvious detail not documented in the HH-suite README.

The `-s` flag sorts the ffindex by key during build, which is convenient
but not the final order (see "load-balancer sort" below).

### Prefilter cs219 derivation (cstranslate)
The column-state prefilter sequences (cs219), which `hhsearch` uses for fast
candidate selection before invoking the expensive Viterbi step, were generated
from the a3m ffindex in a single batch operation:

```
cstranslate -f -x 0.3 -c 4 -I a3m \
  -i db_a3m -o db_cs219
```

`cstranslate` has no single-file mode — it operates on ffindex/ffdata pairs
directly. Flags `-f` (force overwrite), `-x 0.3` (admixture coefficient for
context-state pseudocounts), and `-I a3m` (input format) follow the wiki
specification. The `-c 4` (OpenMP thread count) was chosen empirically:
**cstranslate's memory footprint scales with thread count** (~15 GB base +
~4 GB per thread for an 18 k-entry DB); `-c 4` keeps peak RSS ≈ 31 GB,
within a 96 GB SLURM allocation comfortably. Higher thread counts (`-c 8`)
exceeded a 48 GB cgroup limit in early attempts.

### Load-balancer sort
The HH-suite prefilter splits work across CPU threads by walking
`cs219.ffindex` in order; even load distribution requires the index to be
**sorted descending by entry size** so that long alignments don't bunch at
the end of the queue. To achieve this:

```
sort -k3 -n -r db_cs219.ffindex | cut -f1 > sorting.dat
for triple in cs219 a3m hhm; do
  ffindex_order sorting.dat db_${triple}.ffdata db_${triple}.ffindex \
    db_${triple}.ffdata.ord db_${triple}.ffindex.ord
  mv db_${triple}.ffdata.ord  db_${triple}.ffdata
  mv db_${triple}.ffindex.ord db_${triple}.ffindex
done
```

All three indices are reordered to maintain key alignment.

### Workflow orchestration
Build steps were coordinated with Snakemake v9.20.0 (Mölder et al. 2021)
running on the bwHPC Helix SLURM cluster. Per-protein MSA + HMM construction
was parallelised over a **length-tiered batching scheme** to balance
throughput against SLURM `MaxSubmitPU=1500` and per-job walltime limits:

| Tier   | Length (AA) | Proteins | Per-batch | CPUs | Mem | Walltime | per-protein timeout |
|--------|-------------|----------|-----------|------|-----|----------|---------------------|
| small  | < 800       | 15,850   | 30        | 4    | 24 G | 8 h     | 60 min |
| medium | 800–1500    | 1,785    | 10        | 4    | 24 G | 24 h    | 120 min |
| large  | 1500–3000   | 326      | 4         | 4    | 48 G | 16 h    | 180 min |
| giant  | ≥ 3000      | 46       | 1         | 8    | 64 G | 8 h     | 360 min |
| **total batches:** |     | 18,007   |           |      |     |          |         |

Each batch was executed by a single per-batch shell script
(`workflow/scripts/run_batch.sh`) which iterates over its assigned proteins
serially within the SLURM allocation, with **(i)** an idempotent skip if the
output `.a3m` and `.hhm` already exist (>0 bytes), **(ii)** per-protein
`timeout`-wrapped `hhblits`/`hhmake` invocations, **(iii)** atomic write
semantics (`.tmp → final` only on full success), and **(iv)** a `SIGUSR1`
trap (sent by SLURM 5 min before walltime via `--signal=B:USR1@300`) for
graceful drain.

The `pack_ffindex` rule takes the **intersection** of `{a3m} ∩ {hhm}` —
proteins that permanently failed at one stage are excluded from all three
indices, so the parity check (equal entry counts across a3m, hhm, cs219)
always passes.

### Build history and final coverage
Two successive Snakemake runs were performed:

- **v1 build** (May 12–13, 2026): 17,567 / 18,007 proteins (97.55 % coverage).
  442 long-tail proteins exceeded the initial 20-min per-protein hhblits
  timeout (median failed length: 1,345 AA).
- **v1.1 build** (May 13–15, 2026): tier-aware per-protein timeouts
  (60/120/180/360 min) and per-tier memory bumps (24/24/48/64 GB). All 18,007
  proteins built successfully (100 % proteome coverage). v1.1 supersedes v1.

Validation (see "Validation" below) confirmed that v1.1 returns identical
top hits to v1 for proteins present in both DBs, plus correct (and often
substantially stronger) hits for proteins that were missing in v1.

### Validation
Sensitivity was verified by querying six well-characterised conserved
ribosome-rescue / no-go decay factors (PELO/Q9BRX2, HBS1L/Q9Y450,
NEMF/O60524, LTN1/O94822, ABCE1/P61221, ZNF598/Q86UK7; all human UniProt
sequences) against the final database using `hhsearch`. All six recovered
their expected *Marchantia* orthologs at HH-suite probability 100 with
E-values ranging from 1.1 × 10⁻⁵³ (ZNF598 → Mp5g21820.1) to 2 × 10⁻¹⁵⁷
(NEMF → Mp1g00940.1). The full validation set, expected top hits, and
reference output PDFs are bundled in the end-user repo at
`examples/rqc_batch/` (https://github.com/Papareddy/marchantia_hhdb_user).

### Compute footprint
Approximate wall-clock and CPU consumption on bwHPC Helix `cpu-single`
partition (40 concurrent SLURM jobs, fair-share scheduling):

| Stage | Wall | Cumulative CPU-hours |
|---|---|---|
| Smoke test (10 proteins) | 15 min | < 1 |
| v1 build (msa + hhm, 18 k proteins) | ~4 days | ~3,500 |
| v1.1 retry (failed-batch resubmission) | ~12 h | ~80 |
| Pack stage (cstranslate + ffindex_order, both attempts) | ~12 h | ~50 |
| **Total**                                                 |  ~5 days | **~3,600** |

A3M files account for ~99 % of the on-disk DB footprint (213 GB out of 214 GB).
